## рҹ”— Linked projects (Cohort #1)

This repo is the **template + training-project** for all 4 brownfields in the programme. The 3 pair-projects below are generated from this template W1 Wed PM via the `pair-brownfield-generator` skill, inheriting all 12 debt items + the architecture but reshaping the domain to each pair's federal-acquisitions aspect.

| Repo | Anchor (real federal system) | Domain focus |
|------|------------------------------|--------------|
| [`grants-portal-modern`](https://github.com/KarsunFDE/grants-portal-modern) | **Grants.gov** | Federal grants management вҖ” application intake, peer-review panel, award decisions |
| [`contract-payment-flow`](https://github.com/KarsunFDE/contract-payment-flow) | **WAWF** | Post-award contract administration вҖ” modifications, invoices, closeout |
| [`foia-response-pipeline`](https://github.com/KarsunFDE/foia-response-pipeline) | **FOIA.gov** | FOIA request processing вҖ” intake, triage, redaction, response |

**Task portability**: daily-tasks given against `foia-response-pipeline` apply to **all 4 repos** because the 12 brownfield-debt items + architecture + file-pattern shape are constant. Each pair-project's `domain-mapping.md` (auto-generated by `pair-brownfield-generator` Step 5) is the rename-lookup so a task spec like "Fix audit-log race in `foia-request-service`" auto-translates to each pair's renamed service. See [`training-resources/templates/eod-deliverable.md`](https://github.com/KarsunFDE/training-resources/blob/main/templates/eod-deliverable.md) for the portability-matrix authoring shape.

**Cross-pair integration ladder** (L1 free, L2 required W3-Fri Phase-1-Gate, L3 required W5 AIOps):
- **L1**: shared Auth0 tenant + JWT claims schema (automatic; same across all 4)
- **L2**: each pair-project exposes one `/peer/<resource>` endpoint + consumes one peer's
- **L3**: domain events + W3C `traceparent` propagation across all 3 pair-projects converging at shared OTel Collector

## Brownfield debt вҖ” read this before "fixing" anything

This scaffold has **12 deliberate brownfield-debt items** that the cohort
identifies in `weeks/W01/brownfield-debt.md` on W1 Tue and modernises in
W4 MonвҖ“Thu. **Do not "fix" these before the cohort sees them.**

See `docs/brownfield-debt.md` for the full inventory + which week each item
surfaces. Short list:

1. JWT signature-skip on `/api/public/*` in api-gateway
2. Audit-log written *after* response in foia-request-service (race on crash)
3. No circuit breaker in redaction-review-service вҶ’ foia-request-service
4. No structured-output validation in ai-orchestrator (Bedrock raw passthrough)
5. Pre-v1.0 LangChain `LLMChain(...).run(...)` pattern alongside v1.0 patterns
6. Inconsistent correlation-IDs across services (`X-Request-ID` / `correlationId` / `traceId` / absent)
7. `pinecone-client` listed in `requirements.txt` but never imported
8. Frontend `FoiaRequestListComponent` hardcodes `http://localhost:8081/api/foia-requests` (bypasses gateway)
9. No OWASP input sanitization on foia_request `description` (accepts raw HTML)
10. No multi-tenant boundary вҖ” `agency_id` in schema but no query filter
11. Dockerfiles use `:latest` base-image tags
12. GHA `ci.yml` has `# TODO: re-enable lint when we have time` (linting commented out)

Plus reinforcement gaps: no Postgres volume mount in `docker-compose.yml`; no
service healthchecks; OIDC-to-AWS deploy stub never actually deploys.
