## sec-refresh-token-never-expires

- **Category:** security
- **Inject mode:** edit_existing (file did not exist in baseline → created
  per location_template; fallback search confirmed no TokenService anywhere
  in api-gateway)
- **Location:** `services/api-gateway/src/main/java/com/karsunfde/foiapipeline/gateway/TokenService.java`
- **Test:** `services/api-gateway/src/test/java/com/karsunfde/foiapipeline/gateway/RefreshTokenExpirationDebtTest.java`
- **Difficulty:** medium
- **Scheduled unlock:** W4 (AI Security Engineering — token management theme)
- **Surfaces as:** OWASP A07 (identification + auth failures), FedRAMP AC-12
  (session termination) finding. `TokenService.issueRefresh(...)` mints a JWT
  carrying `typ=refresh` but omits the `exp` claim — token is valid forever
  once issued. Stealing a refresh token (cookie theft, log leak, XSS on the
  requester portal) yields permanent access.
- **Discovery path:** mint a refresh token, parse it back, observe
  `getExpirationTime() == null`. Equivalent finding via `jwt.io` paste.
- **FOIA-domain risk:** the requester perspective is the standard external
  entry point for this system. A leaked requester refresh token gives an
  attacker indefinite read of pending FoiaRequests, ExemptionDeterminations
  in flight, and any ResponsivePages already proposed for redaction.
- **Fix:** add `.expirationTime(Date.from(Instant.now().plus(30, DAYS)))` to
  the `JWTClaimsSet.Builder` chain.
- **Why assessable:** OWASP A07 + FedRAMP AC-12 are concrete rubric anchors;
  the W4 AI-Security day covers token management as a first-class theme.

## sec-sql-string-concat-aspect-search

- **Category:** security
- **Inject mode:** edit_existing (file did not exist in baseline; created at
  the location_template path)
- **Location:** `services/redaction-review-service/src/main/java/com/karsunfde/foiapipeline/redactionreview/repository/SearchRepository.java`
- **Test:** `services/redaction-review-service/src/test/java/com/karsunfde/foiapipeline/redactionreview/repository/SqlStringConcatSearchDebtTest.java` (uses H2 embedded)
- **Difficulty:** easy
- **Scheduled unlock:** W4 (AI Security Engineering — OWASP A03 Injection day)
- **Surfaces as:** classic OWASP A03 Injection. `searchByStatus(String status)`
  builds SQL as `"SELECT * FROM foia_requests WHERE status = '" + status + "'"`.
  Passing `"' OR '1'='1"` returns every row.
- **Discovery path:** call the endpoint with an injection string; observe
  `searchByStatus("' OR '1'='1").size() == totalCount()`. Equivalent finding
  via grep for `+ status +` in repositories.
- **FOIA-domain risk:** the status enum on the redaction-review table is
  `PROPOSED | UNDER_REVIEW | APPROVED_FOR_RELEASE | WITHHELD` — the FOIA
  Officer's view of where each proposed redaction sits in the workflow. An
  injection enumerates pending release decisions across requesters, bypassing
  the Mongo-side per-requester ACL.
- **Fix:** `jdbc.query("SELECT * FROM foia_requests WHERE status = ?", new Object[]{status}, MAPPER)`.
- **Why assessable:** OWASP A03 is the most-cited federal pentest finding;
  mirrors a real FedRAMP MOD finding pattern (KEV-grade).
- **Dependency note:** required `spring-boot-starter-jdbc` + H2 (test scope)
  added to `services/redaction-review-service/pom.xml`. JDBC alongside Mongo
  is a legitimate federal-acquisitions co-existence (Postgres for audit +
  reporting; Mongo for documents).

## rel-retry-without-backoff

- **Category:** reliability
- **Inject mode:** edit_existing (added new `draft()` method to existing
  `AiOrchestratorClient.java`; pre-existing `draftSsdd()` + `factorSuggest()`
  methods untouched)
- **Location:** `services/redaction-review-service/src/main/java/com/karsunfde/foiapipeline/redactionreview/client/AiOrchestratorClient.java`
- **Test:** `services/redaction-review-service/src/test/java/com/karsunfde/foiapipeline/redactionreview/client/RetryWithoutBackoffDebtTest.java`
- **Difficulty:** hard
- **Scheduled unlock:** W5 (AIOps Resilience4j anchor)
- **Surfaces as:** under a transient 503 from ai-orchestrator (or upstream
  Bedrock `ThrottlingException`), `AiOrchestratorClient.draft()` retries 5×
  in a tight loop. Total elapsed time is under 100ms regardless of backend
  state — versus a properly-backed-off retry which sums to 7.5s+ over five
  attempts.
- **Discovery path:** stub the AI service to return 503 5×; observe total
  elapsed time. Tail logs during a load test against `/draft` while
  ai-orchestrator is throttled.
- **FOIA-domain risk:** redaction-proposer batches over multi-thousand-page
  responsive-document uploads fan out hundreds of concurrent reviews. With
  no backoff, every transient throttle from Bedrock fires 5N tight retries
  → AI service takes itself down.
- **Fix:** wrap in Resilience4j `@Retry` with
  `IntervalFunction.ofExponentialBackoff(500ms, 2.0)` and `maxAttempts=5`.
  Mirrors baseline Item 3 (circuit breaker) but at the AI-call boundary
  instead of the Spring-service boundary.
- **Why assessable:** W5 AIOps Resilience4j anchor — direct rubric line.

## obs-error-rethrown-stack-lost

- **Category:** observability
- **Inject mode:** edit_existing → create_new fallback. Pool YAML's
  `location_template` is `services/ai-orchestrator/app/{aspect_primary_entity_singular_lowercase}_service.py`
  (= `app/foia_request_service.py`). That file does not exist in the flat
  baseline ai-orchestrator layout (no per-entity service files in
  acquire-gov). Created fresh per SKILL fallback rule.
- **Location:** `services/ai-orchestrator/app/foia_request_service.py`
- **Test:** `services/ai-orchestrator/tests/test_error_rethrown_stack_lost_debt.py`
- **Difficulty:** medium
- **Scheduled unlock:** W5 (AIOps — Python exception chaining + debugging)
- **Surfaces as:** `draft_for_review(...)` and
  `draft_for_review_with_failure(...)` both catch the underlying
  `botocore.exceptions.ClientError` and re-raise as flat
  `RuntimeError("AI call failed")` — no `from e`. The chained `__cause__`
  is `None`; the original ThrottlingException + stack is invisible to
  debugging.
- **Discovery path:** wrap a deliberate ClientError in the test seam;
  observe `exc.value.__cause__ is None`. Equivalent finding via running the
  redaction-proposer under throttled Bedrock and grepping for stack traces.
- **FOIA-domain risk:** when Bedrock throttles a multi-thousand-page
  ResponsivePages redaction batch, the SRE needs to see ThrottlingException
  in the chained cause to know to dial concurrency back. Stripping the cause
  hides that signal.
- **Fix:** `raise RuntimeError("AI call failed") from e` (note the `from e`).
- **Why assessable:** W5 AIOps debugging discipline; Python exception
  chaining is a foundational PEP-3134 pattern most developers half-know.

## ai-bedrock-no-cost-limit

- **Category:** ai-integration
- **Inject mode:** create_new (file does not exist in baseline)
- **Location:** `services/ai-orchestrator/app/cost_guard.py` (no-op stub —
  reachable via `from app.cost_guard import BedrockCostGuard,
  CostLimitExceeded`, wired into `app/__init__.py`)
- **Test:** `services/ai-orchestrator/tests/test_bedrock_no_cost_limit_debt.py`
- **Difficulty:** hard
- **Scheduled unlock:** W5 (AIOps cost governance + HITL #7 authority boundary)
- **Surfaces as:** `BedrockCostGuard.check(tenant, est_tokens)` is a no-op
  stub. Every Bedrock InvokeModel call in `app/main.py` +
  `app/foia_request_service.py` bypasses any per-tenant ceiling. The
  locked test fires 1000 calls for the same tenant and observes
  zero `CostLimitExceeded` raises.
- **Discovery path:** read `cost_guard.py` — the `check()` body is
  literally `return None`. Equivalent finding via load-testing a single
  tenant against `/draft-foia-request` and watching Bedrock cost spike
  with no rate-limit kick.
- **FOIA-domain risk:** the requester perspective is potentially
  adversarial. An attacker triggers a multi-thousand-page responsive-
  document upload → redaction-proposer fans out → without ceiling, a
  single tenant drains the Karsun-paid Bedrock account (per D-050,
  Cohort #1 runs against a single shared AWS account in W5).
- **Fix:** Redis-backed counter + per-tenant `LIMITS` dict + middleware
  that runs `BedrockCostGuard.check()` before every Bedrock-bound
  endpoint; raise `CostLimitExceeded` as HTTP 429. HITL #7 (W5 Wed AIOps
  auto-remediation authority) gates per-tenant ceiling decisions.
- **Why assessable:** D-050 + D-060 cost discipline; W5 AIOps cost
  governance + HITL #7 authority boundary — direct rubric anchor.
